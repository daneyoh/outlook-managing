# 배포 · 업데이트 방법 (개발자용)

새 버전을 만들어 배포하는 방법과, 이미 설치한 사람들이 업데이트받는 방식을 정리합니다.

---

## 1. 새 버전 내기 (개발자 = 나)

### 1단계 — 코드 수정 후 테스트 통과 확인 → 커밋/푸시
먼저 테스트 안전망을 돌려 초록인지 확인합니다 (분류·경로·설정 로직 회귀 방지):
```
python -m pytest -q
```
통과(예: `111 passed, 1 skipped`)를 확인한 뒤 커밋/푸시:
```
git add -A
git commit -m "수정 내용"
git push
```

### 2단계 — 버전 번호 올리기
`설치/installer.iss` 의 `MyAppVersion` 을 올립니다.
```
#define MyAppVersion "1.0.1"   ; 예: 1.0.0 → 1.0.1
```

> ⚠️ **`AppId` 는 절대 바꾸지 마세요.**
> 이 값이 같아야 새 설치파일이 기존 설치를 **업그레이드(덮어쓰기)** 로 인식합니다.
> 바꾸면 별개 프로그램으로 **중복 설치**됩니다.

### 3단계 — 설치파일 재빌드
```
설치\build_installer.bat
```
- 개인 `config.py` → placeholder 자동 교체 → PyInstaller 빌드 → `config.py` 복원 →
  Inno Setup 컴파일까지 한 번에 처리
- 결과물: `설치\output\OutlookWidgetSetup.exe`
- (Inno Setup 미설치 시 https://jrsoftware.org/isdl.php 에서 먼저 설치)

### 4단계 — GitHub Release 로 발행
```
gh release create v1.0.1 "설치/output/OutlookWidgetSetup.exe#OutlookWidgetSetup.exe" ^
  --title "v1.0.1" ^
  --notes "이번 변경: (무엇이 바뀌었는지 한 줄)"
```
- 태그(`v1.0.1`)는 매번 새 값으로. `MyAppVersion` 과 맞추면 관리가 편합니다.
- 릴리스 노트에 **바뀐 점**을 적어두면 사용자가 업데이트 이유를 압니다.

---

## 2. 이미 설치한 사람들이 업데이트받는 법

**자동 업데이트 기능은 없습니다.** 아래처럼 안내하면 됩니다.

1. 사람들에게 "새 버전 나왔으니 다시 받아 설치하세요" 알림
2. 링크는 **항상 이 고정 링크** (버전 올려도 안 바뀜):
   ```
   https://github.com/daneyoh/outlook-managing/releases/latest
   ```
3. 새 `OutlookWidgetSetup.exe` 실행 → 기존 설치 위에 **덮어쓰기(업그레이드)**

### 사용자 데이터는 유지됩니다
- 로그인 토큰, 할 일·메모·완료 기록, Azure 설정(`user_config.json`)은
  설치 폴더의 `02. DB` 에 있고, 업데이트는 **프로그램 파일만** 교체합니다.
- 따라서 재설치해도 **다시 로그인하거나 Azure ID 를 다시 입력할 필요 없습니다.**

## 요약

| 구분 | 개발자(나) | 사용자(남) |
|------|-----------|-----------|
| 할 일 | 버전 ↑ → `build_installer.bat` → `gh release create` | `/latest` 에서 새 exe 받아 실행 |
| 링크 | 새 태그 `vX.Y.Z` | 항상 `/releases/latest` (고정) |
| 데이터 | — | 로그인·설정·기록 그대로 유지 |

---

## 참고

- 설치파일에 **디지털 서명이 없어** 실행 시 Windows SmartScreen 경고가 뜰 수 있습니다.
  → **추가 정보 → 실행** 으로 진행. 경고 제거는 코드 서명 인증서(유료) 필요.
- 배포 exe 에는 개인 Azure 시크릿이 포함되지 않습니다(빌드 시 placeholder 로 교체).
- `gh` (GitHub CLI) 로그인이 필요합니다: `gh auth login` (최초 1회).
